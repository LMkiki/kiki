# 移车智能助手 AI 开发实现指南
> 本文档用于指导AI完成移车智能助手Web服务的全流程开发，所有功能实现、逻辑判断、交互话术必须严格遵循本文档规则，不得偏离原始需求。

---

## 一、项目核心目标与基础约束
### 1.1 核心目标
建设一个面向移车场景的文本对话Web服务，通过多轮对话完成移车申请信息采集，依据固定业务规则自动判断「受理申请 / 不予受理 / 转人工 / 结束会话」，同时完成全量数据留存，配套后台管理能力。

### 1.2 基础约束（必须严格遵守）
1.  **仅支持纯文本对话**：本版本不实现语音输入、语音识别、语音播报任何语音相关功能。
2.  **核心闭环流程**：用户发起移车诉求 → 系统采集4项必填信息 → 业务规则校验 → 输出最终结果并结束会话。
3.  **数据全量留存**：必须持久化保存完整对话记录、信息采集结果、会话最终状态、会话结束原因，不可丢失任何交互数据。
4.  **后台能力要求**：必须支持对话记录查询、测试用例增删改查管理。
5.  **交互体验要求**：支持用户在一句话中同时提供多项信息，系统自动识别并跳过对应提问环节，无需重复追问。

---

## 二、强制业务规则（优先级最高）
所有逻辑判断必须严格执行以下规则，不得自定义放宽或修改判定标准。

### 2.1 车牌校验规则
1.  仅**正式民用机动车号牌**进入自动移车流程。
2.  车牌格式需符合国内正式车牌编码规则，通过正则完成基础格式校验。
3.  以下情况不进入普通自动移车流程：
    - 非浙江省号牌（车牌首字不是「浙」）
    - 临时号牌
    - 摩托车号牌
    - 军警、消防、救护等特种车辆号牌
    - 无牌车辆

### 2.2 移车原因规则
1.  交通事故类诉求（车被撞、剐蹭、追尾、肇事等）不属于普通移车服务范围。
2.  识别到事故类关键词后，直接提示用户拨打122处理，并结束会话。

### 2.3 地址校验规则
1.  移车地址必须至少明确到**浙江省内的城市+区县**层级。
2.  地址需归属浙江省内；若仅识别到区县，可结合车牌属地补全市级信息，标准化为「浙江省XX市XX区/县」格式。
3.  无法识别到有效区县、地址不在浙江省内，判定为地址无效，引导用户补充；多次补充无效则转人工。

### 2.4 转人工触发规则
满足任意一条，即触发转人工处理并结束自动会话：
1.  用户主动输入「人工、客服、转人工」等明确诉求关键词。
2.  连续3轮对话无法识别出对应环节的有效信息。
3.  用户提供的信息存在明显异常且无法引导修正。

## 三、对话状态机核心实现规范
对话引擎采用状态机模式实现，所有会话流转必须基于状态编码控制，确保逻辑可追溯、可扩展。

### 3.1 状态总览
| 状态编码 | 状态名称 | 状态说明 |
|----------|----------|----------|
| INIT | 初始状态 | 会话创建完成，等待用户发起诉求，发送初始引导语 |
| WAIT_PLATE | 待输入车牌 | 缺失车牌号信息，等待用户提供对方车牌号 |
| CONFIRM_PLATE | 待确认车牌 | 识别到有效车牌，向用户二次确认车牌信息 |
| WAIT_COLOR | 待输入车牌颜色 | 缺失车牌颜色信息，等待用户告知 |
| WAIT_ADDRESS | 待输入移车地址 | 缺失移车地址，等待用户提供详细地址 |
| WAIT_REASON | 待输入移车原因 | 缺失移车原因，等待用户说明 |
| VERIFYING | 全量校验中 | 4项信息收集完成，执行全量业务规则校验 |
| ACCEPTED | 受理成功 | 校验全部通过，受理移车申请，会话结束 |
| REJECTED | 不予受理 | 不符合服务范围，驳回申请，会话结束 |
| TRANSFER_HUMAN | 转人工 | 触发转人工规则，转接人工处理，会话结束 |

### 3.2 状态流转核心逻辑
1.  **一次性完整信息场景**：用户单条消息包含全部4项信息 → 直接进入`VERIFYING`校验 → 校验通过跳转`ACCEPTED`，校验失败跳转对应异常状态。
2.  **分步采集场景**：按「车牌 → 车牌确认 → 车牌颜色 → 地址 → 原因」的顺序追问，每收集并确认一项，自动跳转到下一个缺失项的等待状态。
3.  **异常中断场景**：任意环节命中事故原因、非浙牌、用户要求人工等规则，直接跳转到对应结束状态，终止信息采集流程。

### 3.3 信息抽取要求
1.  必须支持单句多字段识别：从用户文本中同时提取「车牌号、车牌颜色、移车地址、移车原因」四类信息。
2.  增量补全逻辑：每轮对话更新已收集字段，已确认的字段不再重复提问。
3.  容错处理：识别到模糊信息时，优先引导用户确认，而非直接判定无效。

### 3.4 标准话术规范
所有系统回复话术需与需求示例风格保持一致，核心场景固定话术如下：
- 初始引导：`您好，请告知您要移的对方车牌号。`
- 车牌确认：`好的，请再确认一下对方车牌是XX牌照的XXXXXX，对吗？`
- 颜色询问：`好的，请问是蓝牌车，对吗？`
- 地址询问：`好的，请问移车地址在哪个区县，如杭州市西湖区。`
- 原因询问：`好的，最后一个问题，请问移车的原因是什么？`
- 受理成功：`好的，您反馈的在{标准化地址}且车牌号为{车牌号}的{车牌颜色}车，因{移车原因}申请移车，我们马上为您联系车主，稍后通知您移车结果，请保持手机畅通，再见。`
- 事故拒办：`抱歉，您的情况不在114移车服务范围，请直接拨打122进行处理。`
- 转人工：`已为您转接人工客服，请稍候。`

## 四、Web 服务架构设计要求
### 4.1 整体架构分层
采用前后端分离的四层架构，各层职责解耦：
1.  **展示层**：用户对话Web页面 + 后台管理Web页面
2.  **接口层**：C端对话接口 + B端管理接口，统一鉴权、限流、日志
3.  **业务层**：对话引擎（状态机+信息抽取）、业务规则引擎、数据管理模块
4.  **数据层**：MySQL存储结构化持久化数据，Redis缓存活跃会话状态

### 4.2 推荐技术栈
- 后端服务：Python + FastAPI（异步高性能，适配对话场景，开发效率高）
- 用户前端：Vue 3 + 轻量化聊天组件
- 管理前端：Vue 3 + Element Plus
- 数据库：MySQL 8.0
- 缓存：Redis 7
- 部署：Docker + Nginx

### 4.3 核心模块职责
1.  **对话引擎模块**：会话生命周期管理、状态机流转、文本信息抽取、回复话术生成
2.  **规则引擎模块**：车牌校验、属地校验、地址校验、原因校验、异常场景判定
3.  **数据管理模块**：对话记录持久化、移车申请单管理、测试用例CRUD
4.  **后台管理模块**：对话记录多维度查询、测试用例在线管理

## 五、数据库设计规范
必须包含以下4张核心业务表，字段可扩展但不可缺失核心字段。

### 5.1 会话表（session）
存储会话基础信息与最终状态
| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| session_id | varchar(64) | 主键 | 会话唯一标识 |
| status | varchar(32) | 非空 | 会话状态编码，对应状态机 |
| end_reason | varchar(64) | 可空 | 会话结束原因（正常受理/事故拒办/非浙牌/转人工等） |
| created_at | datetime | 非空 | 会话创建时间 |
| ended_at | datetime | 可空 | 会话结束时间 |

### 5.2 对话消息表（message）
存储每一轮交互的完整消息
| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | bigint | 主键自增 | 消息ID |
| session_id | varchar(64) | 非空，外键 | 关联会话ID |
| sender_type | tinyint | 非空 | 发送方：1-用户，2-系统 |
| content | text | 非空 | 消息文本内容 |
| created_at | datetime | 非空 | 消息发送时间 |

### 5.3 移车申请表（move_car_apply）
结构化存储采集完成的移车申请信息
| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | bigint | 主键自增 | 申请ID |
| session_id | varchar(64) | 非空，外键 | 关联会话ID |
| plate_number | varchar(32) | 可空 | 对方车牌号 |
| plate_color | varchar(16) | 可空 | 车牌颜色 |
| address | varchar(255) | 可空 | 标准化后的移车地址 |
| reason | varchar(255) | 可空 | 移车原因 |
| accept_status | tinyint | 非空 | 受理状态：0-未完成 1-已受理 2-不予受理 3-转人工 |
| created_at | datetime | 非空 | 申请创建时间 |

### 5.4 测试用例表（test_case）
存储后台管理的测试用例数据
| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | bigint | 主键自增 | 用例ID |
| case_name | varchar(64) | 非空 | 用例名称 |
| input_content | text | 非空 | 测试输入文本 |
| expect_reply | varchar(255) | 非空 | 预期系统回复 |
| expect_status | varchar(32) | 非空 | 预期会话状态 |
| created_at | datetime | 非空 | 创建时间 |
| updated_at | datetime | 非空 | 更新时间 |
